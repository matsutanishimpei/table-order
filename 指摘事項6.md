# 指摘事項 6: Findy スコア向上のための高度なリファクタリング

本ドキュメントでは、さらなる技術的負債の削減と、Findy 等のコード解析ツールで高く評価される「エンタープライズ品質」への到達を目指した改善案をまとめます。

---

## 1. アーキテクチャの洗練 (Architectural Refinement)

### 1.1 独自例外階層の導入と共通ハンドリングの徹底
- **現状**: `DatabaseException` は存在するが、ビジネスロジックエラー（在庫不足等）とシステムエラー（接続失敗等）が混在している可能性があります。
- **改善案**:
    - `BaseException` (RuntimeException) を頂点とした階層化。
    - `BusinessException`: ユーザーに通知すべき業務エラー（バリデーション失敗、権限不足等）。
    - `SystemException`: ログ出力と管理者通知が必要な致命的エラー。
- **効果**: `BaseServlet` で一括ハンドリングし、エラーの種類に応じた適切なレスポンス（HTTP 400系 vs 500系）を自動生成できます。

### 1.2 サービスレイヤーへのビジネスロジック集約
- **現状**: サーブレット側でリクエストパラメータを詳細に検証し、サービスを呼び出しています。
- **改善案**: サービスメソッドの引数を、DTO または Record に変更。サービス内でも再度（あるいは主導で）ビジネスルールに基づくバリデーションを実行します。
- **効果**: サーブレットは「入出力の制御」に専念し、サービスが「純粋な業務ルール」を保証する「関心の分離」が評価されます。

---

## 2. モダンな Java 機能の活用 (Modern Java Features)

### 2.1 Java 17 `record` の積極的活用
- **現状**: `ValidationResult` や `DailySales` など、データ保持専用のクラスに Lombok が使用されています。
- **改善案**: これらを Java 17 の `record` に置き換えます。
- **効果**: 不変性（Immutability）が言語レベルで保証され、コードがより簡潔かつ意図が明確になります。Findy 等では最新機能の活用として評価されます。

### 2.2 Stream API と関数型プログラミングの徹底
- **現状**: リストの変換やフィルタリングに従来の `for` ループが残っている箇所があります。
- **改善案**: `java.util.stream` を活用し、宣言的な記述に統一します。
- **効果**: コードの可読性と宣言的な記述による「意図の伝わりやすさ」が向上します。

---

## 3. 保守性と堅牢性の向上 (Maintainability & Robustness)

### 3.1 依存性注入 (DI) パターンの統一
- **現状**: サーブレットのデフォルトコンストラクタ内で `new ServiceImpl()` を行っています。
- **改善案**: 軽量なサービスロケーターパターンを導入するか、コンストラクタインジェクションのみを公開し、生成を管理するファクトリクラスを用意します。
- **効果**: ユニットテスト時のモック差し替えが容易になり、結合度の低い設計として評価されます。

### 3.2 Javadoc とドキュメントの充実
- **現状**: 主要なクラスには Javadoc がありますが、private メソッドや例外の説明が不足している場合があります。
- **改善案**: すべてのパブリック API に対して `@param`, `@return`, `@throws` を含む完全な Javadoc を付与します。
- **効果**: メンテナンス性が飛躍的に高まり、チーム開発に適したプロフェッショナルな codebase となります。

---

## 4. セキュリティとパフォーマンス (Security & Performance)

### 4.1 SQL インジェクション対策の静的保証
- **現状**: `PreparedStatement` を使用していますが、SQL 文字列の一部を動的に組み立てている箇所がないか再点検が必要です。
- **改善案**: SQL 文字列の結合を禁止し、すべて `SqlConstants` かプレースホルダ経由に限定するコーディング規約を適用。
- **効果**: セキュリティリスクをゼロに近づけ、安全なアプリケーションとして評価されます。

### 4.2 フロントエンドリソースの最適化
- **現状**: JSP 内にインライン CSS や JS が散見される可能性があります。
- **改善案**: 静的ファイルを外部ファイル化し、キャッシュ戦略を明確にします。
- **効果**: レスポンス速度の向上と、保守性の向上が期待できます。
