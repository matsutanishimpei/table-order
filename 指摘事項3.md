## 🔴 重要（セキュリティ・データ整合性）

### S-1. `login.jsp` の CSRF トークンが未生成のままになる ✅適用済み

**ファイル**: `view/login.jsp` (L34), `filter/AuthFilter.java`  
**問題**: ログイン画面の `<form>` に `csrf_token` を埋め込んでいるが、`/Login` は `AuthFilter` の保護対象外（ログイン前なのでフィルタを通らない）。そのためトークンがセッションに生成されず、`${csrf_token}` は常に空になる。`LoginServlet.doPost()` 側でもトークン検証を行っていないため、**ログインフォームは CSRF 攻撃に対して無防備**である。  
**対処**: `LoginServlet.doGet()` でトークンを生成してセッションにセットするか、ログイン用の CSRF 検証を `doPost()` 内に実装する。

---

### S-2. `CartServlet.doPost()` の `NumberFormatException` がサイレントに握り潰されている ✅適用済み

**ファイル**: `controller/CartServlet.java` (L73-75)  
**問題**: `Integer.parseInt()` が失敗した場合、`catch (NumberFormatException e)` で何もせず握り潰している。悪意のあるユーザーが不正なパラメータを送信した場合に検知できない。  
**対処**: ログ出力を追加し、不正入力を記録する。

```java
} catch (NumberFormatException e) {
    logger.warning("不正な入力値が送信されました: " + e.getMessage());
}
```

---

### S-3. `UserAdminServlet.doPost()` で文字エンコーディング設定が局所的 ✅適用済み

**ファイル**: `controller/UserAdminServlet.java` (L63)  
**問題**: `request.setCharacterEncoding("UTF-8")` を `UserAdminServlet` のみで呼び出している。他の Servlet（`LoginServlet`, `OrderServlet` 等）ではこの処理がないため、日本語を含む POST パラメータが文字化けするリスクがある。  
**対処**: `AuthFilter` または新たなエンコーディングフィルタで `request.setCharacterEncoding("UTF-8")` を一元的に実行し、個別 Servlet からは削除する。

## 🟡 中程度（設計・保守性）

### D-1. `OrderService` インターフェースのインポートが完全修飾名のまま ✅適用済み

**ファイル**: `service/OrderService.java` (L31, L37)  
**問題**: `java.util.List<model.OrderItemView>` と完全修飾名で記述されている。ファイル先頭で `import` しているにもかかわらず、一部のメソッドだけ完全修飾名を使っており、スタイルが不統一。

### D-3. `serialVersionUID` の定義が一部の Servlet に限定されている ✅適用済み

**ファイル**: 全 Servlet  
**問題**: `LoginServlet`, `LogoutServlet`, `CashierServlet`, `AdminHomeServlet` には `serialVersionUID` が定義されているが、他の Servlet（`CartServlet`, `MenuServlet`, `HallServlet` 等）には定義されていない。`HttpServlet` は `Serializable` を実装しているため、全クラスで定義するのが正しい。  
**対処**: 全 Servlet クラスに `private static final long serialVersionUID = 1L;` を追加し、統一する。

---

### D-4. `CartServlet.doPost()` 内の冗長なローカル変数 ✅適用済み

**ファイル**: `controller/CartServlet.java` (L62)  
**問題**: `ProductService service = productService;` と、既にフィールドとして保持している `productService` をわざわざローカル変数に再代入している。リファクタリング前の名残と思われるが、冗長で可読性を下げている。  
**対処**: ローカル変数を削除し、`productService.findById(productId)` を直接呼び出す。

```java
// Before
ProductService service = productService;
Product p = service.findById(productId);

// After
Product p = productService.findById(productId);
```

### Q-5. `CartServlet.doPost()` で `Integer.parseInt()` を使用している ✅適用済み

**ファイル**: `controller/CartServlet.java` (L49-50)  
**問題**: `ValidationUtil.parseIntSafe()` が存在するにもかかわらず、`Integer.parseInt()` を直接使用し `NumberFormatException` を手動で catch している。  
**対処**: `ValidationUtil.parseIntSafe()` に統一する。これにより try-catch ブロック自体が不要になる。

```java
int productId = util.ValidationUtil.parseIntSafe(request.getParameter("productId"), -1);
int quantity = util.ValidationUtil.parseIntSafe(request.getParameter("quantity"), 0);
if (productId < 0 || quantity <= 0) {
    // 不正値の場合は無視
    response.sendRedirect("Menu");
    return;
}
```
