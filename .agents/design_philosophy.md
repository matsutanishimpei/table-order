# Table Order System: 設計思想 (Design Philosophy)

本プロジェクトにおいて、技術的負債を最小化し、エンタープライズ品質の「美しいコード」を維持するための核心となる考え方を定義します。

## 1. 核心的な考え方 (Core Concepts)

- **不変性 (Immutability) の重視**:
  - 状態の変化を予測可能にするため、データモデルには **Java 17 `record`** を全面的に採用します。一度生成されたデータは変更せず、変更が必要な場合は新しいインスタンスを生成するスタイルを推奨します。
- **Null 安全 (Null Safety) の徹底**:
  - `NullPointerException` を排除するため、`null` を返す可能性のあるメソッドは **`Optional<T>`** を戻り値とします。呼び出し側での安全なハンドリングを強制します。
- **宣言的記述 (Declarative Programming)**:
  - 「何をするか（How）」ではなく「何をしたいか（What）」を記述します。ループ処理には **Stream API** を活用し、簡潔で意図が伝わりやすいコードを記述します。

## 2. セキュリティと堅牢性の方針 (Security & Robustness)

- **SQLインジェクションの完全排除（静的保証）**:
  - SQLはすべて `SqlConstants` で定義し、`PreparedStatement` のプレースホルダのみを使用します。コード内での文字列結合によるSQL構築を禁止することで、脆弱性の混入を静的に防ぎます。
- **強力な認証基盤**:
  - パスワードは **BCrypt (with Pepper)** で保護し、タイミング攻撃対策を含めたプロフェッショナルな認証実装を維持します。
- **徹底したリソース管理**:
  - 外部接続（DB, ファイル等）は **`try-with-resources`** を必須とし、例外発生時も確実にクローズされることを保証します。

## 3. 品質と保守性の基準 (Quality Standards)

- **Javadoc = 技術仕様書**:
  - 全ての公開APIに対し、`@param`, `@return`, `@throws` を含む完全な Javadoc を付与します。コードを読むだけで仕様が理解できる状態を維持します。
- **例外の役割分担**:
  - 業務エラー（ユーザーへの通知が必要）とシステムエラー（致命的な障害）を例外クラスで明確に区別し、適切なログ出力とエラー画面への遷移を行います。
- **テストによる品質保証**:
  - 単なる正常系テストに留まらず、JUnit 5 の `@ParameterizedTest` 等を活用して境界値や不正入力を網羅します。外部依存はモック化し、安定したCI実行を可能にします。
- **フロントエンドの最適化**:
  - 画像の遅延読み込み（`loading="lazy"`）や、Tailwind CSS を用いた現代的で一貫性のある UI/UX を提供します。
