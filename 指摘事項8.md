# 指摘事項 8: さらなる進化へのロードマップ (Technical Excellence Roadmap)

前回のフェーズでデータベースとオブザーバビリティの基盤は完璧になりました。ここからは、大規模運用や高い開発速度を維持するための「アーキテクチャの洗練」と「フロントエンドのモダン化」に焦点を当てます。

## 1. キャッシュ戦略によるパフォーマンスの極限化 (Caching Layer)

### 現状と課題
現在はすべてのリクエストが直接データベースに問い合わせを行っています。メニューカテゴリや商品一覧など、頻繁に変更されないデータ（参照系）に対しても毎回クエリが走るため、ユーザー数が増加した際のDB負荷が懸念されます。

### 改善案
- **Caffeine / Redis の導入**: メモリ内キャッシュ（Caffeine）または分散キャッシュ（Redis）を導入し、参照頻度の高いデータをキャッシュします。
- **キャッシュ無効化ロジック (TTL/Eviction)**: 商品更新時にキャッシュを自動でクリアする仕組みをサービス層に実装します。
- **目的**: レスポンスタイムをミリ秒単位で短縮し、DBコストを削減します。

## 2. HTMX / SPA-lite によるユーザー体験の刷新 (Modern UX)

### 現状と課題
現在の JSP ベースの画面遷移は、操作のたびにページ全体がリロードされるため、モダンな Web アプリ（React 等）と比較すると操作感が重く感じられます。しかし、全体を React 等で書き直すのはコストが高すぎます。

### 改善案
- **HTMX の導入**: ページ全体をリロードせずに、必要な部分（カートの更新、ステータスの変更など）だけをサーバーから HTML で取得して置換する HTMX を導入します。
- **目的**: サーバーサイドレンダリング（JSP）の利点を活かしつつ、SPA（Single Page Application）のようなサクサクとした操作感を実現します。

## 3. 非同期処理とイベント通知 (Async Processing & WebSockets)

### 現状と課題
注文が入った際の「キッチンへの通知」などは、現在ポーリング（定期的な再読み込み）または手動更新に頼っています。リアルタイム性が求められる現場では、数秒の遅れがオペレーションに影響します。

### 改善案
- **WebSocket (Jakarta WebSocket) の実装**: 注文が発生した瞬間に、キッチン端末やホール端末へリアルタイムで通知をプッシュします。
- **非同期イベントバス**: 注文完了後の「メール送信」や「集計処理」をメインスレッドから切り離し、非同期で実行します。
- **目的**: リアルタイムな店舗運営を支援し、システムのリソース効率を最大化します。

## 4. API ドキュメンテーションと契約テスト (OpenAPI & Contract Testing)

### 現状と課題
サーブレットが提供するエンドポイントの仕様がコード内に埋もれており、外部連携や将来的なモバイルアプリ対応を行う際の仕様把握が困難です。

### 改善案
- **OpenAPI (Swagger) 仕様書の作成**: 各サーブレットの入出力を OpenAPI 形式で定義し、常に最新のドキュメントを参照可能にします。
- **目的**: 開発者間のコミュニケーションコストを削減し、API の品質を担保します。

## 5. ドメインモデルの独立とクリーンアーキテクチャへの移行 (Domain Centric)

### 現状と課題
現在、ビジネスロジック（Service）が DAO（DB）に強く依存しており、純粋な業務ルールとインフラストラクチャが混在し始めています。

### 改善案
- **ドメインエンティティの分離**: DBのレコード（Record）とは別に、ビジネスルールを内包した「ドメインオブジェクト」を定義します。
- **依存性逆転 (DIP)**: Service が DAO 実装ではなくインターフェースにのみ依存するようにし、ビジネスロジックの単体テストをさらに容易にします。
- **目的**: 10年後も変更に強い、保守性の高いコードベースを構築します。

---
これらのステップは、本プロジェクトを単なる「管理ツール」から、**「モダンな SaaS 級のプラットフォーム」**へと昇華させるためのものです。
