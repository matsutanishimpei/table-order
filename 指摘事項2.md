# 指摘事項2 — シニアエンジニア目線コードレビュー

レビュー実施日: 2026-04-22  
対象: `table-order` プロジェクト全体（Java / JSP / SQL）

---

## 🔴 重大（セキュリティ・データ破損リスク）

### S-1. `database.properties` がGitにコミットされている

**ファイル**: `src/main/resources/database.properties`  
**問題**: DBパスワード（`db.mysql.pass`）とセキュリティの要である Pepper 値（`app.security.pepper`）がリポジトリに平文でコミットされている。`.gitignore` には `src/main/java/database.properties` が記載されているが、**パスが間違っている**（正しくは `src/main/resources/database.properties`）。  
**影響**: リポジトリにアクセスできる全員がDB接続情報とパスワードハッシュの秘密鍵を取得できてしまう。  
**対処**:
1. `.gitignore` のパスを `src/main/resources/database.properties` に修正する
2. `git rm --cached src/main/resources/database.properties` で追跡を解除する
3. Pepper値はコミット済みなので、**本番運用前に必ずローテーション**する

---

### S-2. `KitchenServlet` / `HallServlet` の `doPost` で `NumberFormatException` の可能性

**ファイル**: `controller/KitchenServlet.java` (L34), `controller/HallServlet.java` (L32)  
**問題**: `Integer.parseInt(request.getParameter("itemId"))` が `null` や非数値のリクエストに対してバリデーションなしで呼ばれている。悪意のあるリクエストやブラウザの誤動作で `NumberFormatException`（500エラー）が発生する。  
**対処**: `ValidationUtil.parseIntSafe()` を使用し、不正値の場合はエラーページ or リダイレクトで処理する。

```java
// Before（危険）
int itemId = Integer.parseInt(request.getParameter("itemId"));

// After（安全）
int itemId = ValidationUtil.parseIntSafe(request.getParameter("itemId"), -1);
if (itemId < 0) {
    response.sendRedirect("Home");
    return;
}
```

---

### S-3. `ProductDetail` / `Menu` でも同様の `NumberFormatException`

**ファイル**: `controller/ProductDetail.java` (L23), `controller/Menu.java` (L44)  
**問題**: `Integer.parseInt()` がバリデーションなしで使用されている。URLパラメータを手入力で改ざんされると500エラーになる。  
**対処**: S-2と同様に `ValidationUtil.parseIntSafe()` に置き換える。

---

### S-4. `UserAdminServlet.doPost` のパラメータバリデーション不足

**ファイル**: `controller/UserAdminServlet.java` (L62)  
**問題**: `Integer.parseInt(request.getParameter("role"))` が `null` チェックなしで呼ばれている。`role`パラメータが送信されなかった場合（HTTPリクエスト改ざん時）に500エラーが発生する。  
**対処**: `ValidationUtil.parseIntSafe()` を使用するか、try-catch でガードする。

## 🟡 中程度（設計・保守性）

### D-1. DAO の直接インスタンス化（DI未導入）

**ファイル**: 全 Servlet（例: `CartServlet` L54, `CashierServlet` L28）  
**問題**: `new OrderDAOImpl()` や `new ProductDAOImpl()` など、Servlet 内で実装クラスを直接 `new` している。インターフェースに依存している意味が薄れており、ユニットテスト時にモックに差し替えることができない。  
**対処**（段階的に）:
1. **短期**: コンストラクタ or フィールドで DAO を受け取れるようにし、テスト時に差し替え可能にする
2. **中期**: Service 層をすべての業務で導入し、Servlet → Service → DAO の一貫した呼び出しフローに統一する

---

### D-2. Service 層の適用が部分的

**ファイル**: `service/OrderService.java`, `service/impl/OrderServiceImpl.java`  
**問題**: `OrderService` は導入されているが、キッチン・ホールのステータス更新やユーザー管理など、他の業務ロジックは Servlet から直接 DAO を呼んでいる。Service 層の導入基準が不統一。  
**対処**: トランザクション管理やバリデーションが必要な処理は Service 層を経由するように統一する。特に `UserAdminServlet` の insert/update/delete は `UserService` を作るべき。

---

### D-3. `OrderDAOImpl` の責務過多（God Object傾向）

**ファイル**: `database/impl/OrderDAOImpl.java`（372行）  
**問題**: 注文登録、ステータス更新、未精算テーブル一覧、テーブルステータス、売上集計、商品別ランキングなど、**あらゆる業務の参照系処理**がこの1クラスに集中している。  
**対処**: 責務ごとにDAOを分離する検討を行う。
- `OrderDAO` — 注文の登録・ステータス更新
- `SalesDAO` — 売上集計（`getTotalSales`, `findDailySales`, `findProductSalesRanking`）
- `TableDAO` — テーブル状態（`findAllTableStatus`, `findUnsettledTables`）

---

### D-4. `ResultSet` → Model への変換コードが大量に重複している

**ファイル**: `database/impl/ProductDAOImpl.java` (L29-38, L56-64, L82-90)  
**問題**: `findAll()`, `findByCategory()`, `findById()` の3メソッドで、`ResultSet` から `Product` への変換コードが完全にコピペされている。`OrderDAOImpl` でも `findActiveOrderItems()` と `findReadyOrderItems()` が同構造。  
**対処**: `private Product mapResultSetToProduct(ResultSet rs)` のようなマッピングメソッドを1つ定義して共通化する。

```java
private Product mapRow(ResultSet rs) throws SQLException {
    Product p = new Product();
    p.setId(rs.getInt("id"));
    p.setCategoryId(rs.getInt("category_id"));
    // ...
    return p;
}
```

---

### D-5. `DBManager` でコネクションプーリングが未導入

**ファイル**: `database/DBManager.java` (L53-54)  
**問題**: `DriverManager.getConnection()` を毎回呼び出しており、リクエストごとに TCP 接続の確立と切断が発生する。同時アクセスが増えると性能劣化やコネクション枯渇の原因になる。  
**対処**: `HikariCP` や Tomcat の `DataSource`（JNDI）を利用したコネクションプーリングを導入する。

---

### D-6. `User.role` のマジックナンバー

**ファイル**: `model/User.java` (L26, L34, L42, L50), `controller/UserAdminServlet.java` (L62)  
**問題**: `role == 1`, `role == 2` ... のように数値リテラルで権限を判定している。`OrderConstants` でステータス定数を定義した設計と一貫性がない。  
**対処**: `Role` 列挙型（Enum）を定義する、または最低限 `UserConstants` クラスに定数として定義する。

```java
public class UserConstants {
    public static final int ROLE_ADMIN   = 1;
    public static final int ROLE_KITCHEN = 2;
    public static final int ROLE_HALL    = 3;
    public static final int ROLE_CASHIER = 4;
    public static final int ROLE_USER    = 10;
}
```

## 🟢 軽微（コード品質・可読性）

### Q-1. `BaseDAO` インターフェースが未活用

**ファイル**: `database/BaseDAO.java`  
**問題**: `BaseDAO<T>` を定義して `CategoryDAO`, `ProductDAO`, `UserDAO` が継承しているが、`OrderDAO` は継承していない。また `BaseDAO` の `findAll()` しかメソッドがなく、汎用的な CRUD 基盤として活用されていない。`import` されているが使われていない DAO実装クラスもある。  
**対処**: `BaseDAO` をプロジェクトの設計方針に合わせて拡充するか、過剰な抽象化であれば削除して個別の DAO インターフェースに統一する。

---

### Q-2. `CartItem` の `@AllArgsConstructor` がフィールド数と合わない可能性

**ファイル**: `model/CartItem.java`  
**問題**: `@AllArgsConstructor` は全フィールドを引数にとるコンストラクタを生成する。`CartItem` には `productId`, `name`, `unitPrice`, `quantity` の4つのフィールドがあるが、`getSubtotal()` はカスタムメソッドとして計算している。以前のコンストラクタ `CartItem(int, String, int, int, int)` は5引数だったため、既存の呼び出し元がある場合はコンパイルエラーになる可能性がある。  
**対処**: 既存の全呼び出し元を確認し、問題なければ現状維持。問題がある場合はカスタムコンストラクタを明示的に定義する。

---

### Q-3. Servlet のクラス名と URL パターンの命名規則が不統一

**ファイル**: 全 Servlet  
**問題**:
- 一部は動詞/名詞そのまま（`Login`, `Menu`, `Logout`, `OrderHistory`）
- 一部は `〜Servlet` サフィックス（`CartServlet`, `KitchenServlet`, `CashierServlet`）  
  
命名規則が混在しており、可読性と検索性が下がっている。  
**対処**: いずれかに統一する（推奨: `〜Servlet` サフィックス）。例: `LoginServlet`, `MenuServlet`

---

### Q-4. `Menu.java` でセッション確認が `AuthFilter` と二重になっている

**ファイル**: `controller/Menu.java` (L28-32)  
**問題**: `AuthFilter` がすでに `/Menu` を保護対象として設定しているにも関わらず、`Menu.doGet()` 内でも `session == null` チェックを行っている。二重チェック自体は壊れないが、保守時に「どこで認証しているのか」が不明瞭になる。  
**対処**: `AuthFilter` に任せて、Servlet 側のセッション確認コードは削除する。

---

### Q-5. `OrderDAOImpl.findAllTableStatus()` のビジネスロジック混在

**ファイル**: `database/impl/OrderDAOImpl.java` (L262-288)  
**問題**: ステータスコードからラベル文字列（`"調理待ち"`, `"配膳待ち"` 等）への変換処理が DAO 内に書かれている。これは表示のためのロジック（プレゼンテーション層の責務）であり、DAO の責務ではない。  
**対処**: DAO は生のステータスコードのみ返し、ラベル変換は Service 層か JSP 側で行う。

---

### Q-6. `e.printStackTrace()` の残存

**ファイル**: `database/DBManager.java` (L43, L74)  
**問題**: `e.printStackTrace()` は標準エラー出力にのみ出力され、ログファイルに記録されない。本番環境ではログが見落とされる可能性が高い。  
**対処**: `java.util.logging.Logger` を使用したログ出力に統一する。

---

### Q-7. CSRF トークンがセッション中に固定されている

**ファイル**: `filter/AuthFilter.java` (L42-46)  
**問題**: CSRF トークンはセッション開始時に1度だけ生成され、セッションが続く限り同じ値が使い回される（Synchronizer Token Pattern としては最低限機能するが）。理想的にはリクエストごとにトークンをローテーションし、使用済みトークンを無効化する。  
**対処**: セキュリティ要件に応じて、リクエストごとのトークン再生成を検討する。

---

### Q-8. `CashierServlet.doPost` で会計失敗時のエラーハンドリングなし

**ファイル**: `controller/CashierServlet.java` (L50-56)  
**問題**: `service.completeCheckout(tableId)` の戻り値 `boolean` を確認していない。会計処理が失敗しても、ユーザーには正常に画面遷移してしまい、失敗に気づけない。  
**対処**: 戻り値を確認し、`false` の場合はエラーメッセージを表示する。

```java
boolean success = service.completeCheckout(tableId);
if (!success) {
    request.setAttribute("error", "会計処理に失敗しました。");
    doGet(request, response);
    return;
}
```
