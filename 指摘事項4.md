# 製品品質に関する指摘事項 (Cloudinary連携および全体)

今回の Cloudinary 連携の実装および現在のプロジェクト状態に基づき、製品品質（保守性、セキュリティ、信頼性、パフォーマンス）の観点からさらなる改善が望まれる事項をまとめます。

## 1. セキュリティ (Security)

### 1.1 認証情報の外部管理
- **現状**: Cloudinary の `API_SECRET` が `database.properties` に直接記述されています。
- **リスク**: このファイルが誤って Git 等のバージョン管理システムにコミットされると、認証情報が漏洩し、不正利用（画像の削除やストレージの占有）を招く恐れがあります。
- **対策**: 環境変数（OSの環境変数や `System.getenv()`）から読み込む形式に変更するか、Git 管理外の外部設定ファイルとして分離を徹底すべきです。

### 1.2 アップロード時のファイルバリデーション 【適用済み】
- **現状**: サーブレット側でファイルサイズの上限（10MB）は設定されていますが、ファイル形式（MIMEタイプ）の厳密なチェックが行われていません。
- **リスク**: 画像以外の不適切なファイルがアップロードされたり、悪意のあるスクリプトを含むファイルがアップロードされるリスクがあります。
- **対策**: `util.ValidationUtil.isValidImage()` を実装し、`image/jpeg`, `image/png`, `image/webp`, `image/gif` のみを許可するロジックを適用しました。

## 2. 信頼性とエラーハンドリング (Reliability)

### 2.1 外部サービス依存のリスク管理 【適用済み】
- **現状**: Cloudinary へのアップロードが失敗した場合、`null` が返り、DBには反映されませんが、ユーザーへのエラー通知が「更新失敗」という曖昧なものになる可能性があります。
- **リスク**: ネットワーク障害や Cloudinary の障害時に、原因の特定が遅れます。
- **対策**: `CloudinaryUtil` で詳細なログを記録し、サーブレット側でアップロード失敗を検知してユーザーに具体的なエラーメッセージを表示するよう適用しました。

### 2.2 データ整合性 (Cloudinary vs DB) 【適用済み】
- **現状**: 商品が削除されたり、画像が上書きされた際、Cloudinary 上の古い画像がそのまま残る（孤立する）可能性があります。
- **リスク**: ストレージ容量を無駄に消費し、長期的にはコスト増につながります。
- **対策**: `CloudinaryUtil.deleteByIdentifier()` を実装し、新しい画像のアップロード成功時に古い画像を Cloudinary から自動削除するロジックを適用しました。

## 3. パフォーマンス (Performance)

### 3.1 画像の遅延読み込み (Lazy Loading) 【適用済み】
- **現状**: `menu.jsp` 等で大量の商品画像が表示されます。
- **リスク**: 画面遷移時にすべての画像を一斉に読み込もうとし、初期表示が重くなる可能性があります。
- **対策**: 各画面の `<img>` タグに `loading="lazy"` 属性を付与し、ブラウザの最適化機能を活用するように適用しました。

## 4. 保守性 (Maintainability)

### 4.1 public_id の管理 【適用済み】
- **現状**: DB にはフル URL を保存し、表示時にパスを解析して変換パラメータを挿入していました。
- **リスク**: 将来的に配信ドメインが変更されたり、Cloudinary 以外のストレージに移行する際、既存の全データを書き換える必要が出てきます。
- **対策**: DB に `v[version]/[public_id]` 形式の識別子のみを保存し、表示時に `CloudinaryUtil` で URL を動的に組み立てる設計を適用しました。

## 5. テスト (Testing)

### 5.1 外部APIのモックテスト 【適用済み】
- **現状**: JUnit テストにおいて Cloudinary 連携部分のテストが不足していました。
- **リスク**: 開発環境での意図しない API 消費や、オフライン環境でのビルド失敗が発生します。
- **対策**: `ImageStorageProvider` インターフェースを導入して画像操作を抽象化し、`ProductAdminServlet` に依存性注入（DI）を行うことで、Mockito を使用したモックテスト（`ProductAdminServletTest`）を構築しました。
