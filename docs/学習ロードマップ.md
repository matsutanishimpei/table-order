# コードリーディング学習ロードマップ

## このロードマップの目的

この教材は、MVC と Java Servlet の基礎を学び終えた人が、複数の画面とデータベースを持つWebアプリケーションのコードを一連の流れとして理解するためのガイドです。

この順番でコードを1周すると、次の内容を説明できるようになります。

- HTTPリクエストがFilter、Servlet、Service、DAOを通ってDBへ届く流れ
- JSPへ表示データを渡す方法と、POST後にリダイレクトする理由
- セッションにログイン情報とカートを保存する仕組み
- Controller、Service、DAOに処理を分ける理由
- 複数のDB更新をトランザクションでまとめる理由
- インターフェースとコンストラクタ注入がテストを容易にする仕組み
- 認証、認可、CSRF、パスワードハッシュが守っているもの

目安時間は、コードを読みながら進めて3〜5時間です。最初からすべてのクラスを読むのではなく、注文のライフサイクルを軸に必要なファイルだけを辿ります。

## 最初に持っておく全体像

```text
ブラウザ
  ↓ HTTPリクエスト
Filter（文字コード・認証・CSRF・ログ・セキュリティヘッダー）
  ↓
Controller / Servlet（入力と画面遷移）
  ↓
Service（業務ルールとトランザクション）
  ↓
DAO（SQLの実行）
  ↓
MySQL

Servlet
  ↓ request属性
JSP（HTMLの生成）
  ↓ HTTPレスポンス
ブラウザ
```

先に [クラス図](クラス図.md) と [ER図](er_diagram.md) を眺めてください。この時点では暗記せず、「注文は `orders` と `order_items` に分かれている」「ServiceがControllerとDAOの間にある」と分かれば十分です。

## 1. プロジェクトの入口を確認する

最初に次のファイルを読みます。

1. `pom.xml`
2. `src/main/webapp/index.jsp`
3. `src/main/java/controller/LoginServlet.java`
4. `src/main/webapp/WEB-INF/view/login.jsp`

`pom.xml`では、Java 17、Servlet API、JSTL、MySQL、JUnit、Mockitoを探します。すべての設定を理解する必要はありません。アプリ本体の依存関係とテスト用の依存関係が、`scope`によって分けられている点を確認してください。

次に、`index.jsp`から`/Login`へ移動し、`LoginServlet#doGet`がログイン用JSPへforwardする流れを追います。`doPost`では以下を順番に確認します。

- リクエストパラメータからIDとパスワードを取得する
- 入力値を検証する
- `UserService#login`を呼び出す
- 成功時にセッションIDを変更する
- `User`をセッションへ保存する
- ロールに応じた画面へredirectする

### 理解チェック

- `forward`と`redirect`では、ブラウザのURLとリクエスト回数がどう変わるでしょうか。
- ログイン成功後に`request.changeSessionId()`を呼ぶのはなぜでしょうか。
- `UserService`をServlet内で直接`new`せず、`ServiceFactory`から取得する利点は何でしょうか。

対応するテストは`src/test/java/controller/LoginServletTest.java`です。本体を読んだ後で、成功・失敗・ロール別遷移をどのようにモックしているか確認します。

## 2. メニュー表示でMVCの基本を復習する

次の順に読みます。

1. `src/main/java/controller/MenuServlet.java`
2. `src/main/java/service/ProductService.java`
3. `src/main/java/service/impl/ProductServiceImpl.java`
4. `src/main/java/database/ProductDAO.java`
5. `src/main/java/database/impl/ProductDAOImpl.java`
6. `src/main/java/database/SqlConstants.java`の`PRODUCT_`から始まる定数
7. `src/main/webapp/WEB-INF/view/menu.jsp`

`MenuServlet#doGet`を起点に、カテゴリーIDが以下のように流れる様子を追ってください。

```text
クエリパラメータ categoryId
  → MenuServlet
  → ProductService
  → ProductDAO
  → PreparedStatementのプレースホルダー
  → productsテーブル
```

取得結果は逆向きに`List<Product>`として戻り、Servletが`request.setAttribute`でJSPへ渡します。JSPでは`c:forEach`、`c:if`、`c:out`がそれぞれ何をしているか確認します。

### 理解チェック

- ServletがSQLを直接実行しないのはなぜでしょうか。
- `Product`が`record`で定義されていることには、どんな利点がありますか。
- JSPで商品名を`${product.name}`のまま出力せず、`c:out`を使う理由は何でしょうか。

対応するテストは`MenuServletTest`、`ProductServiceImplTest`、`ProductDAOImplTest`です。同じ機能を層ごとにテストしている点に注目してください。

## 3. カートでセッションと不変オブジェクトを理解する

次のファイルを読みます。

1. `src/main/java/controller/CartServlet.java`
2. `src/main/java/model/CartItem.java`
3. `src/main/webapp/WEB-INF/view/menu.jsp`内のカート送信フォーム
4. `src/test/java/controller/CartServletTest.java`

カートはDBではなく、`HttpSession`内の`List<CartItem>`として保持されます。商品追加時に、既存商品の数量を直接変更せず、新しい`CartItem`を作ってリスト内の要素を置き換えている箇所を確認してください。これは`record`が不変であるためです。

また、フォームから送られた商品名や価格を信用せず、商品IDを使ってServiceから商品を再取得している点も重要です。ブラウザから送られる値は利用者が変更できるため、サーバー側の情報を正とします。

### 理解チェック

- カートをセッションに置く場合、複数の利用者のカートが混ざらないのはなぜでしょうか。
- 商品価格までフォームのhidden項目で受け取る設計には、どんな危険がありますか。
- `CartItem`が通常の可変クラスだった場合、数量更新のコードはどう変わるでしょうか。

## 4. 注文確定でService層とトランザクションを理解する

このロードマップの中心です。次の順に読みます。

1. `src/main/java/controller/OrderServlet.java`
2. `src/main/java/service/OrderService.java`
3. `src/main/java/service/impl/OrderServiceImpl.java`の`createOrder`
4. `src/main/java/database/TransactionManager.java`
5. `src/main/java/database/OrderDAO.java`
6. `src/main/java/database/impl/OrderDAOImpl.java`
7. `src/main/java/database/JdbcExecutor.java`
8. `src/main/java/database/SqlConstants.java`の`ORDER_`から始まる定数

まず`OrderServlet`の責務を確認します。Servletが行うのは、セッションから利用者とカートを取り出すこと、入力条件を確認すること、Serviceを呼ぶこと、結果に応じて画面遷移することです。

業務処理は`OrderServiceImpl#createOrder`にあります。次の処理が1つのトランザクションに含まれます。

1. 各商品を`FOR UPDATE`付きで取得し、販売可能か確認する
2. DB上の最新の商品名と価格で注文明細を組み直す
3. テーブルに未会計の注文があるか確認する
4. 必要なら`orders`へ親レコードを登録する
5. 検証済みの内容を`order_items`へ一括登録する
6. 監査ログを登録する
7. 全処理が成功した場合だけcommitする

途中で例外が発生すると、`TransactionManager`がrollbackします。たとえば、`orders`だけ登録できて`order_items`の登録に失敗した状態を残さないためです。

### 理解チェック

- 商品確認と注文明細登録を同じトランザクションに含めるのはなぜでしょうか。
- `Connection`をDAOメソッドへ渡している箇所と、DAO自身が接続を取得する箇所では何が違いますか。
- Serviceが`false`を返す場合と、例外を投げる場合をどのように使い分けていますか。
- カートへ追加した後に商品価格が変わった場合、どの価格が`order_items`へ保存されますか。

対応するテストは`OrderServiceImplTest`、`TransactionManagerTest`、`OrderDAOImplTest`です。特に、途中失敗時のテストが何を保証し、モックだけでは実際のrollbackをどこまで保証できないかを考えてください。

## 5. 厨房・配膳・会計で状態遷移を追う

注文後は、1件の`order_items.status`が次のように変わります。

```text
STATUS_ORDERED（注文済み）
  → STATUS_COOKING_DONE（調理完了）
  → STATUS_SERVED（配膳済み）
  → STATUS_PAID（会計済み）
```

次のファイルを順に読みます。

1. `src/main/java/model/OrderConstants.java`
2. `src/main/java/controller/KitchenServlet.java`
3. `src/main/java/controller/HallServlet.java`
4. `src/main/java/service/impl/OrderServiceImpl.java`の`updateItemStatus`
5. `src/main/java/controller/CashierServlet.java`
6. `OrderServiceImpl`の`completeCheckout`
7. `src/main/java/database/impl/OrderDAOImpl.java`の状態更新メソッド

`updateItemStatus`では許可された状態遷移だけを受け付けます。さらに、更新SQLの`WHERE`句へ現在の状態を含め、別の処理が先に更新していた場合に上書きしないようにしています。

会計は複数の注文明細と親注文を更新するため、再びトランザクションを使います。未配膳の商品がある場合に会計を拒否する処理も確認してください。

### 理解チェック

- 状態を単なる整数の大小だけで判定せず、許可された遷移を明示する利点は何でしょうか。
- Serviceで現在状態を確認した後、SQLでも現在状態を条件にするのはなぜでしょうか。
- 会計処理で`orders`と`order_items`の片方だけが更新されると、何が起きますか。

## 6. Filterを通して横断的な処理を理解する

ここまでのリクエストは、Servletへ到達する前後でFilterを通っています。

1. `src/main/java/filter/EncodingFilter.java`
2. `src/main/java/filter/AuthFilter.java`
3. `src/main/java/filter/LoggingFilter.java`
4. `src/main/java/filter/SecurityHeaderFilter.java`
5. `src/main/java/util/CsrfUtil.java`
6. `src/main/java/controller/BaseServlet.java`

各Filterについて、「すべてのServletに必要か」「特定URLだけに必要か」を意識して`@WebFilter`のURLパターンを確認します。`AuthFilter`では、ログイン確認、POSTのCSRF検証、ロールによる認可を1か所にまとめています。

`BaseServlet`は未処理例外をログに残し、利用者には一般的なエラーメッセージだけを表示します。内部例外を画面へ出さない理由も考えてください。

### 理解チェック

- 認証と認可は何が違いますか。
- CSRFトークンはセッションとフォームのどちらに保存され、どう比較されますか。
- ログへ残す情報と、エラー画面へ表示する情報を分ける理由は何でしょうか。
- 複数のFilterが同じURLに適用される場合、順序が処理へ与える影響はありますか。

## 7. テストから設計を逆向きに読む

最後に、次の3種類のテストを比較します。

- Controller: `src/test/java/controller/OrderServletTest.java`
- Service: `src/test/java/service/impl/OrderServiceImplTest.java`
- DAO: `src/test/java/database/impl/OrderDAOImplTest.java`

Controllerテストでは`OrderService`をモックし、HTTP入力と画面遷移に集中します。ServiceテストではDAOをモックし、業務ルールに集中します。DAOテストでは実際のSQLとDBの対応を確認します。

コンストラクタが2つあるServletにも注目してください。引数なしコンストラクタはServletコンテナが使用し、引数付きコンストラクタはテストからモックを渡すために使用します。

### 理解チェック

- Controllerテストで実DBを使わない利点は何でしょうか。
- DAOをすべてモックしたServiceテストだけでは発見できない不具合は何でしょうか。
- `verify`は戻り値のassertionと何が違いますか。

## 8. 1周後の仕上げ課題

コードを変更する前に、紙やMarkdownへ次の内容を書き出してください。

1. `/Order`へのPOSTから`order_items`登録までのシーケンス
2. `CartItem`がブラウザ、セッション、Service、DAOの間をどう移動するか
3. 注文処理中に商品が販売停止された場合の保護方法
4. 厨房担当者がいきなり配膳済みへ変更できない仕組み
5. Controller、Service、DAOそれぞれへ新しい処理を置く判断基準

その後、次の小さな変更から1つ選んで実装すると理解を確認できます。

- カート数量の上限をServiceまたはControllerで検証し、テストを追加する
- 注文失敗時の理由を利用者向けに分類する
- 注文履歴へステータス表示を追加する
- 商品検索条件を1つ追加し、ServletからSQLまで値を通す
- 許可されていない状態遷移のテストケースを追加する

変更後は`mvn clean verify`を実行します。テストだけでなく、Checkstyle、SpotBugs、JaCoCo、依存関係チェックまで通ることを確認してください。

## 読み終えたかの最終チェック

次の問いへコードを指しながら答えられれば、1周完了です。

- MVCにServiceとDAOを加えた各層の責務を説明できる
- GETとPOST、forwardとredirectの使い分けを説明できる
- request、session、DBへ保存するデータの違いを説明できる
- 注文処理にトランザクションが必要な理由を説明できる
- インターフェースとコンストラクタ注入がテストに与える効果を説明できる
- 認証、認可、CSRF、XSS、セッション固定対策がそれぞれ何を防ぐか説明できる
- 仕様変更時に、どの層とどのテストを変更するか判断できる
