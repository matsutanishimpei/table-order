# 運用基礎の学習ガイド

## このガイドの位置づけ

このガイドは、[コードリーディング学習ロードマップ](学習ロードマップ.md)を終えた人が、アプリケーションを継続して安全に動かすための基礎を学ぶものです。

運用機能は次の場所へ分離しています。注文処理などのコアコードを理解するときに、最初から読む必要はありません。

```text
src/main/java/database/DatabaseMigration.java    DB変更の適用
src/main/java/util/AppConfig.java                設定値の解決
src/main/java/controller/HealthCheckServlet.java 稼働確認
src/main/java/filter/LoggingFilter.java           リクエスト追跡
src/main/resources/db/migration/                  バージョン付きSQL
src/test/java/service/OrderFlowSmokeTest.java     代表フローの確認
.github/workflows/ci.yml                          自動検証と設定例
```

## 教材としての設計原則と対象外

このリポジトリは、実運用で使われる技術を無条件に増やすのではなく、MVCとServletを学んだ人がコード全体を1周できる見通しを優先します。機能を採用するときは、次の原則で判断します。

1. 最初に`Servlet → Service → DAO → DB`の流れを追えること
2. 運用機能はコアの業務コードから分離できること
3. 同じ目的の設定やファイルを複数の場所へ置かないこと
4. 新しい概念には、実コード、テスト、説明資料をセットで用意すること
5. 網羅性より、代表的な1本を最後まで理解できることを優先すること

次の仕組みは不足や実装漏れではなく、教材の見通しを守るために意図的に導入していません。

| 導入していないもの | 現在の代替 | 導入しない理由 |
|---|---|---|
| ブラウザを使う完全なE2E基盤 | Serviceから実MySQLまでの注文スモークテスト | Tomcatやブラウザドライバーの起動設定が、MVCの学習量を上回るため |
| Prometheusなどのメトリクス基盤 | `/health`、構造化ログ、リクエストID | 複数サービスの構築と運用が別教材相当になるため |
| Docker ComposeやKubernetes | CIのMySQLサービスとTestcontainers | コンテナ構成がアプリケーション構造の理解を妨げるため |
| 高機能な設定・秘密情報管理製品 | `AppConfig`と環境変数 | 製品固有の設定より、秘密情報を分離する原則を先に学ぶため |
| DIコンテナや大規模フレームワーク | インターフェース、コンストラクタ注入、`ServiceFactory` | オブジェクト生成と依存関係をJavaコードとして読める状態を保つため |

これらが常に不要という意味ではありません。次の条件が生じたときに、別の発展教材または実運用プロジェクトとして導入を検討します。

- JSPを含む画面回帰が頻発し、Controllerテストでは防げなくなった
- 複数サーバーの性能、エラー率、遅延を継続的に観測する必要が生じた
- DB、アプリケーション、外部サービスを同じ手順で配備する必要が生じた
- 秘密情報の更新、監査、権限分離を組織的に管理する必要が生じた
- 手作業の依存生成が増え、`ServiceFactory`の見通しが悪くなった

発展機能を追加するときも、既存の学習ロードマップを壊さず、独立したディレクトリ、プロファイル、または別教材へ分離する方針です。

## 1. FlywayでDB変更を管理する

DBはJavaコードと同じように、いつ何を変更したかを履歴として管理する必要があります。このプロジェクトではFlywayを使用し、`src/main/resources/db/migration`のSQLをファイル名の順に適用します。

現在のマイグレーションは次の2つです。

- `V1__create_schema.sql`: テーブルを作成する
- `V2__seed_learning_data.sql`: 学習用の初期データを登録する

FlywayはDB内の`flyway_schema_history`へ適用済みバージョンを記録します。起動時に`DBManager`が`DatabaseMigration#migrate`を呼び、未適用分だけを実行します。

たとえば商品へ表示順を追加する場合、適用済みの`V1`を編集せず、次のファイルを追加します。

```text
src/main/resources/db/migration/V3__add_product_display_order.sql
```

```sql
ALTER TABLE products
    ADD COLUMN display_order INT NOT NULL DEFAULT 0;
```

適用済みファイルを書き換えると、環境ごとに同じバージョンの内容が異なります。修正が必要な場合も、新しいバージョンを追加します。

### 理解チェック

- `schema.sql`を毎回手動実行する方法には、どんな問題がありますか。
- `flyway_schema_history`は何を記録しますか。
- 適用済みの`V1`を編集してはいけないのはなぜですか。

## 2. 秘密情報をコードから分離する

`AppConfig`は設定を次の順番で解決します。

```text
環境変数 → database.properties → デフォルト値
```

ローカルでは`database.properties`を使えるため、学習開始時の手順は単純なままです。本番環境やCIでは環境変数を使い、パスワードやAPIキーをGitやWARへ含めません。

| プロパティ | 環境変数 | 内容 |
|---|---|---|
| `db.is_oracle` | `DB_IS_ORACLE` | Oracleを使用するか |
| `db.mysql.url` | `DB_MYSQL_URL` | MySQL接続URL |
| `db.mysql.user` | `DB_MYSQL_USER` | DBユーザー |
| `db.mysql.pass` | `DB_MYSQL_PASS` | DBパスワード |
| `app.security.pepper` | `APP_SECURITY_PEPPER` | パスワード保護用Pepper |
| `cloudinary.cloud_name` | `CLOUDINARY_CLOUD_NAME` | Cloudinary名 |
| `cloudinary.api_key` | `CLOUDINARY_API_KEY` | Cloudinary APIキー |
| `cloudinary.api_secret` | `CLOUDINARY_API_SECRET` | Cloudinary秘密鍵 |

環境変数名はプロパティ名を大文字にし、`.`を`_`へ置き換えたものです。CIにはテスト専用値の例があります。実運用ではGitHub ActionsのSecretsなどから渡します。

### 理解チェック

- `.gitignore`へ設定ファイルを登録していても、環境変数を使う利点は何ですか。
- PepperをDBパスワードと別に管理する理由は何ですか。
- テストと本番で秘密情報を共用してはいけないのはなぜですか。

## 3. 代表フローをスモークテストする

`OrderFlowSmokeTest`は実際のService、DAO、MySQLを使って、次の1本を確認します。

```text
注文登録 → 調理完了 → 配膳済み → 会計完了
```

単体テストが各クラスの責務を詳しく確認するのに対し、スモークテストは主要部品をつないだときに代表ルートが壊れていないことを短く確認します。

ブラウザ自動操作やTomcatの別プロセス起動は、学習用リポジトリの見通しを大きく損なうため導入していません。これは完全なE2Eではなく、ServiceからDBまでの軽量スモークテストです。

### 理解チェック

- このスモークテストでは、どのような画面の不具合を発見できませんか。
- 単体テストをすべてスモークテストへ置き換えると、何が困りますか。
- 将来ブラウザE2Eを追加するなら、最初の1本はどの操作にしますか。

## 4. ヘルスチェックで稼働状態を確認する

`GET /health`は、アプリケーションからDB接続を取得できるか確認します。

```json
{"status": "UP", "database": "OK"}
```

異常時はHTTP 503と`DOWN`を返します。ロードバランサーや外部監視は、このURLを定期的に確認することで異常を検知できます。

このエンドポイントはDBへ接続できることだけを確認します。注文処理や外部画像サービスの状態までは保証しません。

### 理解チェック

- 異常時にHTTP 200ではなく503を返す理由は何ですか。
- ヘルスチェック内で重い集計SQLを実行すべきでないのはなぜですか。

## 5. リクエストIDでログを追跡する

`LoggingFilter`はリクエストごとに短いIDを発行し、MDCを使ってログへ付与します。同じ値を`X-Request-ID`レスポンスヘッダーにも設定します。

問い合わせ時は`X-Request-ID`をログから検索し、関係する処理を絞り込めます。処理後にMDCを必ず削除するのは、Servletコンテナがスレッドを再利用したときに別リクエストへIDを漏らさないためです。

ログにはパスワード、Pepper、APIキーなどの秘密情報を書いてはいけません。

### 理解チェック

- リクエストIDがない場合、同時アクセスのログ解析はどう難しくなりますか。
- MDCを`finally`で削除する理由は何ですか。
- リクエスト本文をそのままログへ記録する危険は何ですか。

## 6. CIでまとめて確認する

`mvn clean verify`では、単体テスト、MySQL統合テスト、スモークテスト、カバレッジ、Checkstyle、SpotBugs、依存関係スキャンを実行します。

変更をPRへ出す前に、次を確認してください。

1. DB変更には新しいFlywayファイルがある
2. 秘密情報がソースやログへ追加されていない
3. 代表フローのスモークテストが成功する
4. `/health`の意味を変更した場合は監視手順も更新する
5. 新しいログに調査価値があり、秘密情報を含まない

ここまで説明できれば、機能を作る視点だけでなく、変更を安全に届けて問題を調査するための基礎まで1周できています。
